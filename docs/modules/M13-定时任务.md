# 模块13：定时任务

**职责**：每天凌晨由外部调度触发，统一执行所有需要定期处理的业务逻辑

## 调度方案

服务部署在阿里云 FC（Serverless），无内置 cron 能力，采用**外部调度 + HTTP 端点**方案：

- **端点**：`POST /cron/daily`
- **调度**：阿里云 EventBridge 定时规则，每天凌晨 00:00 调用
- **鉴权**：暂不加鉴权，后续可通过请求头密钥或 IP 白名单保护
- **执行策略**：子任务按顺序依次执行，单条记录失败不影响其他记录继续处理，错误收集后统一返回
- **幂等**：所有子任务必须支持重复调用不产生副作用，以应对重试和手动补跑场景

## 子任务清单

### 子任务 1：超时订单处理

> 来源：M06-兑换交易

**目的**：核销码超过有效期（10天）仍未核销或未完全核销的订单，自动取消/结算并退回积分和库存

#### 1a. 完全未核销 → 自动取消

| 项目 | 说明 |
|------|------|
| 扫描条件 | `RedeemOrder.status = PENDING_VERIFY` 且 `verifyCodeExpiresAt < now()` |
| 处理（事务内） | 1. 订单状态改为 `CANCELLED` |
|  | 2. 退还全部大RF/小RF积分（写 IN 流水，reason "超时未核销退还"） |
|  | 3. 恢复全部 SKU 库存（写库存流水） |
|  | 4. 写消息通知 `ORDER_CANCELLED_TIMEOUT` |
| 幂等 | 仅处理 `PENDING_VERIFY` 状态的订单，已取消的自动跳过 |

#### 1b. 部分核销 → 自动结算未核销部分

| 项目 | 说明 |
|------|------|
| 扫描条件 | `RedeemOrder.status = PARTIAL_VERIFIED` 且 `verifyCodeExpiresAt < now()` |
| 处理（事务内） | 1. 遍历所有明细行，找出 `verifiedQuantity < quantity` 的未完全核销行 |
|  | 2. 计算每行未核销数量 `unverified = quantity - verifiedQuantity`，退还对应积分（写 IN 流水，reason "部分核销超时退还"） |
|  | 3. 恢复未核销数量对应的 SKU 库存（写库存流水） |
|  | 4. 将明细行的 `verifiedQuantity` 设为 `quantity`（标记该行已处理完毕） |
|  | 5. 订单状态改为 `VERIFIED` |
|  | 6. 写消息通知 `ORDER_PARTIAL_TIMEOUT`（含已核销商品、退还积分数） |
| 幂等 | 仅处理 `PARTIAL_VERIFIED` 状态的订单，已变为 VERIFIED 的自动跳过 |

> **设计说明**：部分核销的订单不走 CANCELLED，因为用户已经拿到了部分商品。将未核销部分的积分和库存退还后，订单以 VERIFIED 结束，前端可通过明细行的退还流水展示哪些商品因超时被退回。

### 子任务 2：注册周年积分清零

> 来源：M03-积分账户

**目的**：会员注册满一年时，将大RF和小RF积分全部清零

| 项目 | 说明 |
|------|------|
| 扫描条件 | `MemberProfile.activatedAt` 的月日 = 今天，且已满一年（即 `activatedAt + 1年 <= now()`） |
| 处理（事务内） | 1. 该会员所有余额 > 0 的 PointsAccount 清零 |
|  | 2. 每个账户写一条 EXPIRE 流水（reason "注册周年积分清零"） |
|  | 3. 写消息通知 `POINTS_EXPIRED`（含清零的大RF/小RF积分数） |
| 注意 | 清零后会员等级不降级 |
| 幂等 | 检查当天是否已有该账户的 EXPIRE 流水，有则跳过 |

---

### 子任务 3：积分清零提醒

> 来源：M03-积分账户、M09-消息通知

**目的**：提前 3 个月、2 个月、1 个月提醒会员积分即将清零，促使尽快使用

| 项目 | 说明 |
|------|------|
| 扫描条件 | `MemberProfile.activatedAt` 的周年日距今天分别剩余约 90 天、60 天、30 天的会员 |
| 处理 | 写消息通知 `POINTS_EXPIRY_WARN`（含清零日期、剩余天数、当前余额） |
| 前提 | 该会员至少有一个 PointsAccount 余额 > 0（余额为 0 的不提醒） |
| 幂等 | 同一会员 + 同一清零周期 + 同一提醒档位（90/60/30天），查 Message 表是否已有同 type + bizUuid 的记录，有则跳过 |

> bizUuid 建议拼接为 `{memberUuid}_{周年年份}_{档位天数}`，如 `abc123_2027_90`

### 子任务 4：核销码即将过期提醒

> 来源：M06-兑换交易、M09-消息通知

**目的**：核销码到期前 2 天提醒用户尽快到店核销

| 项目 | 说明 |
|------|------|
| 扫描条件 | `RedeemOrder.status` 为 `PENDING_VERIFY` 或 `PARTIAL_VERIFIED`，且 `verifyCodeExpiresAt` 在未来 2 天内（含今天后天之间） |
| 处理 | 写消息通知 `VERIFY_CODE_EXPIRING`（含商品名、过期日期） |
| 幂等 | 同一订单只发一次，查 Message 表是否已有同 type + bizUuid(orderUuid) 的记录 |

---

### 子任务 5：优惠券即将过期提醒

> 来源：M08-优惠券、M09-消息通知

**目的**：券到期前 3 天提醒用户尽快使用

| 项目 | 说明 |
|------|------|
| 扫描条件 | `UserCoupon.status = UNUSED` 且到期时间在未来 3 天内 |
| 到期时间取值 | 优先取 `UserCoupon.expiresAt`；若为空则回退到 `CouponTemplate.validTo` |
| 处理 | 写消息通知 `COUPON_EXPIRY_WARN`（含券名称、过期日期） |
| 幂等 | 同一张券只发一次，查 Message 表是否已有同 type + bizUuid(couponUuid) 的记录 |

---

### 子任务 6：过期优惠券状态更新

> 来源：M08-优惠券

**目的**：将超过有效期的未使用优惠券状态自动更新为已过期

| 项目 | 说明 |
|------|------|
| 扫描条件 | `UserCoupon.status = UNUSED` 且到期时间 `< now()` |
| 到期时间取值 | 优先取 `UserCoupon.expiresAt`；若为空则回退到 `CouponTemplate.validTo` |
| 处理 | 将 status 更新为 `EXPIRED` |
| 幂等 | 仅处理 UNUSED 状态，已过期的自动跳过 |

## 执行顺序与依赖

子任务按以下顺序执行，确保依赖关系正确：

```
1. 超时订单处理            ← 先取消/结算过期订单、退积分（含 PENDING_VERIFY 全额取消 + PARTIAL_VERIFIED 部分退还）
2. 注册周年积分清零        ← 再执行清零（退回的积分也会被清零，如果恰好是周年日）
3. 积分清零提醒           ← 清零完再发提醒（避免已清零的还收到提醒）
4. 核销码即将过期提醒      ← 独立，无依赖（覆盖 PENDING_VERIFY + PARTIAL_VERIFIED）
5. 优惠券即将过期提醒      ← 独立，无依赖
6. 过期优惠券状态更新      ← 放最后，先发提醒再改状态
```

## 接口设计

| 接口 | 方法 | 路径 | 鉴权 | 说明 |
|------|------|------|------|------|
| 每日定时任务 | POST | `/cron/daily` | 暂无（后续加密钥） | 外部调度每天 00:00 调用 |

### 返回格式

```json
{
  "code": 0,
  "message": "ok",
  "data": {
    "startedAt": "2026-03-25T00:00:01.000Z",
    "finishedAt": "2026-03-25T00:00:03.500Z",
    "tasks": [
      { "name": "cancelTimeoutOrders", "processed": 3, "skipped": 0, "failed": 0, "detail": "cancelled: 2, partialSettled: 1" },
      { "name": "clearAnniversaryPoints", "processed": 1, "skipped": 0, "failed": 0 },
      { "name": "pointsExpiryWarn", "processed": 5, "skipped": 2, "failed": 0 },
      { "name": "verifyCodeExpiringWarn", "processed": 2, "skipped": 0, "failed": 0 },
      { "name": "couponExpiryWarn", "processed": 1, "skipped": 0, "failed": 0 },
      { "name": "expireCoupons", "processed": 4, "skipped": 0, "failed": 0 }
    ],
    "errors": []
  }
}
```

每个子任务返回 processed（处理数）、skipped（幂等跳过数）、failed（失败数），失败的详情放 errors 数组里，便于排查。

## 运维与监控

| 项目 | 说明 |
|------|------|
| 日志 | 每次调用记录完整的执行日志（Winston），包含各子任务耗时和处理明细 |
| 失败处理 | 单条记录失败不中断整体执行，错误收集后统一返回 + 记录日志 |
| 手动补跑 | 所有子任务幂等，可随时手动调 `POST /cron/daily` 补跑，不会重复处理 |
| 超时 | 阿里云 FC 函数超时建议设为 300 秒（5 分钟），正常执行应在 30 秒内完成 |

## 依赖关系

- **被依赖**：无（由外部调度触发）
