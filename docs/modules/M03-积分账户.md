# 模块03：积分账户

**职责**：多类型积分的全生命周期管理——录入、消费、过期清零、流水记录

## 积分类型定义

| 类型 | 名称 | 积分来源 | 归属对象 | 关联等级 | 商品池 |
|------|------|----------|----------|----------|--------|
| 大RF积分 | 消费积分 | 线下消费金额，工作人员手动录入（1元=1积分） | 会员本人 | 是（驱动会员等级升降） | 大RF商品 |
| 小RF积分 | 上课积分 | 小朋友上课表现，工作人员手动录入 | 绑定的小朋友 | 否（无等级体系） | 小RF商品 |

> 两种积分完全不互通，各自独立余额、独立流水、独立清零。

## 业务规则

### 积分录入

1. 积分唯一来源是后台工作人员手动录入，系统无自动获取渠道
2. 大RF积分：根据消费金额录入
3. 小RF积分：根据上课表现录入，具体积分数由工作人员自行判断
4. 录入时无需关联消费凭证，系统自动记录操作日志（谁、什么时候、给谁录入了多少积分）
5. 录入错误时可更正，无需审批流程，二次确认后即可修改，操作记入日志

### 积分消费

1. 用户兑换商品时，系统自动扣除对应类型的积分
2. 扣减不影响会员等级（会员等级按大RF累计获得积分计算）
3. 实物商品兑换后生成“待核销”记录，用户线下到前台领取，工作人员核销后状态变为“已兑换”
4. 若会员绑定多个小朋友，则每个小朋友拥有**独立**的小RF积分账户及流水
5. 使用小RF积分兑换商品时，必须指定使用哪个小朋友的积分，仅能扣减该小朋友账户，不可在小朋友之间互相挪用或合并消费

### 积分清零

1. 积分有效期按会员注册周年日计算（注册满一年触发清零）
2. 大RF积分和小RF积分使用同一清零周期，在同一时间点一起清零
3. 清零时按会员维度将该会员的可用积分全部清零（不按批次拆分）
4. 清零前提前3个月、2个月、1个月弹出系统提示
5. 后台可随时查看每个临近积分清零的会员

### 积分流水

每条流水记录包含：积分类型、变动类型（收入/支出/过期）、变动数额、变动原因、操作时间、操作人（如有）

### 记账与一致性设计（建议方案）

1. **余额不手改，只记流水**：积分余额由流水累计得出，禁止直接改余额字段，避免账不平
2. **原子扣减**：兑换扣积分时使用事务（校验余额 + 扣减 + 写流水一次完成），防止并发下超扣
3. **幂等控制**：录入积分、兑换扣减、清零任务都要有业务唯一号，重复请求不重复入账
4. **余额下限**：任何场景都不允许积分余额为负数；若扣减时余额不足，直接失败并提示
5. **可审计**：每笔流水必须能追溯来源单据（录入单/兑换单/清零批次）和操作人

### 积分更正设计（建议方案）

1. 更正仅允许通过“新增一笔反向流水”实现，不修改历史流水原记录
2. 更正时必须填写原因，保留“更正前余额/更正后余额”
3. 更正需二次确认（已确认），不走审批流
4. 更正后若影响大RF等级，触发等级重算

### 清零任务设计（建议方案）

1. 清零任务按日定时执行（建议凌晨）
2. 清零前提醒批次与清零批次使用同一套到期计算逻辑，避免提醒与实际清零不一致
3. 清零执行后生成清零批次号，便于回溯和报表统计
4. 清零失败支持重试，重试需幂等（同一批次不重复清零）

### 定时任务方案

服务部署在阿里云 FC（Serverless），无内置 cron，采用**外部调度 + HTTP 端点**方案：

1. **单一端点** `POST /cron/daily`，内部按顺序执行所有定时子任务
2. 由阿里云 EventBridge 定时规则每天凌晨 00:00 调用该端点
3. 端点暂不加鉴权，后续可通过内部密钥或 IP 白名单保护
4. 每个子任务单条记录失败不影响其他记录继续执行，错误收集后统一返回

**子任务 1：超时订单自动取消（完全未核销）**

- 条件：`status = PENDING_VERIFY` 且 `verifyCodeExpiresAt < now()`
- 处理（事务内）：订单改 CANCELLED → 退还大RF/小RF积分（写 IN 流水，source `REFUND`，reason "超时未核销退还"）→ 恢复 SKU 库存（写库存流水）→ 写消息 `ORDER_CANCELLED_TIMEOUT`
- 幂等：仅处理 PENDING_VERIFY 状态，已取消的不会重复处理

**子任务 2：超时订单结算（部分核销）**

- 条件：`status = PARTIAL_VERIFIED` 且 `verifyCodeExpiresAt < now()`
- 处理（事务内）：退还未核销明细的积分（写 IN 流水，source `REFUND`）和库存 → 已核销部分保留 → 订单状态改 VERIFIED → 写消息 `ORDER_PARTIAL_TIMEOUT`
- 幂等：仅处理 PARTIAL_VERIFIED 状态，已完成的不会重复处理

**子任务 3：注册周年积分清零**

- 条件：`memberProfile.activatedAt` 的月日 = 今天，且已满一年
- 处理（事务内）：该会员所有余额 > 0 的积分账户清零，每个账户写一条 EXPIRE 流水（source `ANNUAL_EXPIRE`，reason "注册周年积分清零"）→ 写消息 `POINTS_EXPIRED`
- 幂等：检查当天是否已有该账户的 EXPIRE 流水，有则跳过

**子任务 4：积分清零提醒**

- 条件：会员 `activatedAt` 距下一个周年日剩余 90 / 60 / 30 天，且大RF或小RF余额 > 0
- 处理：写消息 `POINTS_EXPIRY_WARN`（含剩余天数、当前余额等信息）
- 幂等：同一会员 + 同一清零周期 + 同一提醒档位（90/60/30天），检查是否已有同类消息，有则跳过

**子任务 5：核销码即将过期提醒**

- 条件：订单 `status` 为 PENDING_VERIFY 或 PARTIAL_VERIFIED，且 `verifyCodeExpiresAt` 距今 ≤ 2 天
- 处理：写消息 `VERIFY_CODE_EXPIRING`
- 幂等：同一订单只发一次到期提醒

**子任务 6：优惠券即将过期提醒**

- 条件：优惠券状态为未使用（`UNUSED`），且到期时间距今 ≤ 3 天
- 处理：写消息 `COUPON_EXPIRY_WARN`
- 幂等：同一张券只发一次过期提醒

**子任务 7：过期优惠券状态更新**

- 条件：优惠券状态为 `UNUSED`，且已过到期时间
- 处理：批量将 status 改为 `EXPIRED`
- 说明：不写消息，仅刷状态，防止过期券仍可使用
