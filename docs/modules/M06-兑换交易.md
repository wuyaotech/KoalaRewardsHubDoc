# 模块06：兑换交易（含核销）

**职责**：用户从"想要"到"下单"再到"线下核销"的完整兑换闭环（大RF/小RF积分通用）

> 原 M07-核销 已合并至此模块。兑换下单后生成核销码、线下核销履约属于同一订单生命周期，代码实现中也由 `orders` + `verify` 两个模块协同完成，共享同一张 `RedeemOrder` 表。

## 能力清单

| # | 能力 | 端 | 描述 |
|---|------|---|------|
| 1 | 心愿单 | 小程序 | 收藏感兴趣的商品，类似收藏夹 |
| 2 | 购物车 | 小程序 | 积分足够的商品可加入购物车 |
| 3 | 兑换下单 | 小程序 | 确认兑换，扣减对应类型积分，生成订单与核销码 |
| 4 | 核销码展示 | 小程序 | 用户在小程序出示核销码（二维码 + 8位数字码）给前台 |
| 5 | 扫码/输码核销 | PC | 前台扫码或手动输入数字码完成核销 |
| 6 | 核销状态同步 | 双端 | 核销完成后用户端同步显示"已完成" |
| 7 | 兑换历史 | 小程序 | 查看个人兑换记录和状态（区分大RF/小RF） |
| 8 | 兑换记录查询 | PC | 按用户/积分类型/商品/时间段查看 |
| 9 | 核销历史 | PC | 按关键词查看已核销订单列表 |

## 业务规则

### 商品操作（心愿单 / 购物车）

- 单个商品支持两种操作，**无先后关系**：
  - **加入心愿单**：类似收藏夹，收藏感兴趣的商品
  - **加入购物车**：将商品加入购物车后统一结算
- 心愿单中商品下架后标记为"已下架"，不可操作

### 购物车

- 仅积分足够的商品可加入购物车
- 大RF商品和小RF商品在购物车中分组展示
- 购物车无数量上限
- 小程序购物车包含「正常商品」与「失效商品」两个视图；失效商品进入独立的「失效」Tab 展示
- **失效商品定义**：
  - 商品已下架（`Product.status != ON_SALE`）
  - 商品已删除（`Product.deletedAt != null`）
  - 购物车关联的 SKU 已被删除 / 被新规格替换（`ProductSku.deletedAt != null`），例如后台编辑商品后移除了原规格组合
- **SKU 删除对购物车的影响**：
  - 若后台保存商品时，某个 SKU 规格组合未变化，则复用原 SKU，用户购物车中的该商品不受影响
  - 若后台保存商品时，某个 SKU 被删除或被新的规格组合替换，则不直接清空用户购物车，而是保留原购物车项并在小程序中展示为「失效商品」
  - 失效商品不可勾选结算、不可继续切换幼崽等正常购买操作，用户可删除后重新选择新规格
- **售罄不算失效**：
  - SKU 库存为 0 时，商品仍留在正常购物车，不进入「失效」Tab
  - 售罄商品展示「已售罄」提示，不可勾选结算，不可继续增加数量；恢复库存后可继续参与结算
- **小RF商品必须关联幼崽**：
  - 加购时默认关联当前商城页选中的幼崽
  - 购物车中每件小RF商品显示"兑换惊喜给 XXX"，有多个幼崽时可点击切换
  - 不同小RF商品可以分别关联不同的幼崽
  - 小RF积分仅从该商品关联的幼崽账户扣减，不可跨幼崽合并

### 兑换下单

- 支持混合下单，大RF和小RF商品可同时结算，系统分别扣减对应类型的积分
- 同一订单内的小RF商品可以分别关联不同的幼崽，按明细行各自扣减对应幼崽的小RF积分
- 积分扣减不影响会员等级
- 兑换下单在事务内完成：创建订单 → 扣积分 → 扣库存 → 写积分流水 → 生成核销码；**仅「购物车结算」路径**在成功后还会软删除购物车中已勾选结算的行。**「立即兑换」直兑路径**不经过购物车、不落库购物车，因此事务内不会清理购物车
- 兑换成功后自动生成核销码（8位数字码），有效期 **10 天**

**下单入口与接口（实现约定）**

| 场景 | 小程序行为 | 后端接口 | 说明 |
|------|------------|----------|------|
| 购物车多商品/多行结算 | 勾选商品 → 确认兑换 | `POST /orders/create`，body 含 `cartItemUuids[]`，可选 `childUuid` 兜底 | 与购物车行绑定；成功后在同一事务内软删除对应购物车项 |
| 商品详情立即兑换 | 选规格/数量/幼崽 → 确认兑换 | `POST /orders/create-direct`，body 含 `items: [{ skuUuid, quantity, childUuid? }]`，可选顶层 `childUuid` 兜底 | **单次请求**完成校验与建单，避免先 `cart/add` 再下单的链路与数量合并问题 |

两条接口共用同一套业务校验（余额、每人每月商品限购、库存等）与同一套订单/核销数据模型；**限购统计**以会员当月已存在订单明细（订单非「已取消」）的件数合计为准，与从购物车还是直兑进入无关。

**立即兑换（跳过购物车）**：
- 大RF商品：商品详情页点"立即兑换"→ 弹出选规格/数量 → 结算页 → **一次请求**确认兑换（`create-direct`）
- 小RF商品：商品详情页点"立即兑换"→ 弹出选规格/数量与幼崽 → 结算页 → **一次请求**确认兑换（`create-direct`）

### 核销流程

```
用户兑换成功 → 生成核销码（订单级 + 明细级） → 用户到前台出示 → 工作人员扫码/输码 → 逐项核销 → 全部核销后订单完成
```

- **核销终端**：前台使用 Pad/手机扫码优先，PC后台手动输码作为兜底
- **核销凭证**：下单时生成 1 个订单级核销码 + 每个明细行各 1 个独立核销码（8位数字码，互不重复）
- **核销粒度**：按明细行核销，工作人员扫描/输入的是**明细级核销码**；每次核销可指定数量，支持同一明细行分多次核销
- **不可撤销**：核销后状态不可逆，已核销如有问题走线下处理，由管理员手动重新发放积分

### 兑换订单状态

```
待核销(PENDING_VERIFY) → 部分核销(PARTIAL_VERIFIED) → 已完成(VERIFIED)
待核销(PENDING_VERIFY) → 已取消(CANCELLED)  ← 超时未核销，定时任务自动取消
部分核销(PARTIAL_VERIFIED) → 已完成(VERIFIED)  ← 超时后未核销部分自动退还，已核销部分视为完成
```

| 状态 | 枚举值 | 触发条件 |
|------|--------|---------|
| 待核销 | PENDING_VERIFY | 下单后初始状态，尚无任何明细被核销 |
| 部分核销 | PARTIAL_VERIFIED | 至少有一个明细行已核销部分或全部数量，但尚有未核销的明细 |
| 已完成 | VERIFIED | 所有明细行均已全部核销（`verifiedQuantity >= quantity`），或部分核销超时后未核销部分已自动退还 |
| 已取消 | CANCELLED | 超时未核销自动取消（仅 PENDING_VERIFY，定时任务触发） |

- **部分核销**：订单包含多件商品时，工作人员可逐件扫码核销；核销过程中订单处于 `PARTIAL_VERIFIED` 状态，直到所有明细行都核销完毕才变为 `VERIFIED`
- **不支持用户主动取消**：兑换成功后不可撤销，积分已扣不退
- **超时取消（完全未核销）**：`PENDING_VERIFY` 状态超过 10 天未核销 → 定时任务自动取消（`CANCELLED`），全额恢复库存并回退积分
- **超时结算（部分核销）**：`PARTIAL_VERIFIED` 状态且核销码过期 → 仅退还未核销明细的积分和库存，已核销部分保留 → 订单状态改为 `VERIFIED`，写消息通知 `ORDER_PARTIAL_TIMEOUT`

> 注意：代码中没有独立的"已过期"状态，`CANCELLED` 仅由定时任务超时触发。前端可根据 `verifyCodeExpiresAt` 在展示层区分"已过期"标签。

### 兑换记录

- 每个用户的兑换历史在后台都可见
- 也可以集中不按用户看，查看加总兑换数据，以便调节商品结构

## 数据模型（与 Prisma 对齐）

### RedeemOrder 兑换订单

| 字段 | 类型 | 说明 |
|------|------|------|
| uuid | String | 主键 |
| memberUuid | String | 下单会员 |
| status | RedeemOrderStatus | PENDING_VERIFY / PARTIAL_VERIFIED / VERIFIED / CANCELLED |
| verifyCode | String | 订单级核销码（8位），用于展示和搜索 |
| verifyCodeExpiresAt | DateTime | 核销码过期时间（下单后 +10 天） |
| verifiedAt | DateTime? | 全部核销完成时间（所有明细行核销完毕时写入） |
| operatorUuid | String? | 最后一次核销操作人 |
| totalBigRfPoints | Int | 本单消耗大RF积分总额 |
| totalSmallRfPoints | Int | 本单消耗小RF积分总额 |

> `childUuid` 已从订单级下沉到明细行级（见 RedeemOrderItem），支持同一订单内不同明细行关联不同幼崽。

### RedeemOrderItem 订单明细

| 字段 | 类型 | 说明 |
|------|------|------|
| uuid | String | 主键 |
| orderUuid | String | 所属订单 |
| skuUuid | String | SKU |
| productUuid | String | 商品（冗余，便捷查询） |
| childUuid | String? | 关联的幼崽（小RF商品必填，大RF商品为空） |
| pointsType | PointsType | BIG_RF / SMALL_RF |
| pointsPrice | Int | 单件所需积分（下单时从 Product.pointsPrice 快照） |
| quantity | Int | 兑换数量 |
| pointsAmount | Int | 合计积分 = price × quantity |
| verifyCode | String? | 明细级核销码（8位），工作人员实际扫码/输码用的是这个 |
| verifyCodeExpiresAt | DateTime? | 核销码过期时间（与订单级一致），若为空则以订单级为准 |
| verifiedQuantity | Int | 已核销数量（默认 0），支持分次核销直到 = quantity |

> 核销时工作人员扫描的是**明细级 verifyCode**，系统根据 `verifiedQuantity` 判断该行是否已全部核销。当所有明细行 `verifiedQuantity >= quantity` 时，订单状态变为 `VERIFIED`。

### CartItem 购物车

| 字段 | 类型 | 说明 |
|------|------|------|
| uuid | String | 主键 |
| userUuid | String | 购物车所属用户 |
| skuUuid | String | 具体 SKU |
| childUuid | String? | 关联的幼崽（小RF商品必填，大RF商品为空） |
| quantity | Int | 购买数量 |

> 购物车中同一 SKU + 同一幼崽视为同一行（唯一约束：`userUuid + skuUuid + childUuid`）。同一 SKU 给不同幼崽分别加购会产生多行。
>
> 购物车项默认不因为「SKU 被替换」而立即删除：当商品仍存在、但原 SKU 被后台编辑为失效规格时，`CartItem` 会继续保留，并通过关联到已软删除的 `ProductSku` 在小程序「失效」Tab 中展示，便于用户感知并手动清理。
