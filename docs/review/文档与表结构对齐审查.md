# 文档与表结构对齐审查

> 审查日期：2026-03-24  
> 审查范围：M01~M13 需求文档 vs `packages/service/prisma/schema.prisma`

---

## 一、总体结论

文档与 Prisma schema 在主体结构上高度一致，核心表（User、MemberProfile、PointsAccount、Product、RedeemOrder 等）字段对齐良好。以下列出发现的问题及处理结果。

---

## 二、已解决的问题

### 2.1 ✅ RedeemOrder PARTIAL_VERIFIED 状态 + 明细级核销

**问题**：M06 文档只有 3 种订单状态，schema 有 4 种（含 PARTIAL_VERIFIED）；RedeemOrderItem 有 verifyCode / verifyCodeExpiresAt / verifiedQuantity 字段未文档化。

**处理**：已更新 M06 文档，完整描述了部分核销机制：
- 下单时每个明细行生成独立核销码
- 工作人员扫描明细级核销码逐项核销
- 订单状态：PENDING_VERIFY → PARTIAL_VERIFIED → VERIFIED
- 数据模型补充了 RedeemOrderItem 的 verifyCode、verifyCodeExpiresAt、verifiedQuantity 字段

---

### 2.2 ✅ UserCoupon 补 expiresAt 字段

**问题**：M13 定时任务引用 `UserCoupon.expiresAt` 但 schema 没有该字段。

**处理**：
- Schema 已补 `expiresAt DateTime? @db.DateTime(0)` 字段及索引
- Cron 代码已更新：优先取 `UserCoupon.expiresAt`，若为空回退到 `CouponTemplate.validTo`
- M08 文档补充了 CouponTemplate 和 UserCoupon 数据模型
- M13 文档更新了子任务 5/6 的扫描条件说明

---

### 2.3 ✅ SKU 移除 pointsPrice 预留字段

**问题**：M05 文档说"SKU 不维护积分"，但 schema 有 `ProductSku.pointsPrice` 字段。

**处理**：
- Schema 已移除 `ProductSku.pointsPrice`（经确认代码中无任何读写）
- 保留 `ProductSku.image`（SKU 专属图片，有实际用途）
- M05 文档已同步更新

---

### 2.4 ✅ 其他已确认项（无需改动）

| 问题 | 结论 |
|------|------|
| 未激活会员的积分清零周期 | 不影响——未激活会员没有积分可清零，`activatedAt` 为空时定时任务自动跳过 |
| 解绑小朋友后小RF积分处理 | 不处理——解绑小朋友时软删账户，不写流水，保持现有逻辑 |
| 积分更正是否关联原流水 | 不需要——更正通过新增反向流水实现，不要求关联原单 |
| 券类订单超时取消逻辑 | 券就是一种商品——订单超时取消时一并取消退积分，与普通商品一致（已实现） |

## 三、待定问题（需进一步讨论）

### ~~3.1 CartItem 缺 childUuid~~ → 已确认方案

### ~~3.2 RedeemOrder.childUuid 只能关联一个小朋友~~ → 已确认方案

**确认方案**：`childUuid` 下沉到明细行级别。

根据 UI 设计稿确认：
- 购物车中不同小RF商品可以分别关联不同的幼崽（如"小考拉"、"小草莓"），购物车内可切换
- 商品详情页"立即兑换"时也可选择幼崽

**改动点**（文档已更新，schema/代码待实施）：
- `CartItem` 加 `childUuid`，唯一约束改为 `userUuid + skuUuid + childUuid`
- `RedeemOrderItem` 加 `childUuid`（小RF必填，大RF为空）
- `RedeemOrder.childUuid` 可保留作为冗余快捷字段，但不再作为唯一来源
- 下单扣积分时按每个 item 的 `childUuid` 分别找对应幼崽的小RF积分账户扣减

---

### ~~3.3 数据字典（DictionaryType / DictionaryItem）~~ → 已删除

已确认不需要，代码中无任何引用。Schema 中 DictionaryType / DictionaryItem 两个 model 已移除。性别等选项直接用前端枚举/后端 enum 处理。

---

### 3.4 🔵 M02 姓名字段风格（单字段 vs firstName + lastName）

**现状**：文档写"姓名"一个字段，schema 用 `firstName`（名）+ `lastName`（姓）两字段。

**影响**：中文场景下拆分增加前后端拼接复杂度，但 schema 已全面使用，改动成本较高。

**建议**：保持现状（firstName + lastName），文档备注即可。
