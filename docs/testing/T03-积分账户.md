# 测试：M03-积分账户

> 对应需求：[M03-积分账户](../modules/M03-积分账户.md)

## 一、积分录入（PC端）

### 1.1 大RF积分录入

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 正常录入 | 选择会员 → 输入消费金额 100 元 → 确认 | 大RF积分增加 100，生成一条 IN 流水 | |
| 2 | 录入 0 积分 | 输入金额为 0 | 应拒绝，提示积分必须大于 0 | |
| 3 | 录入负数 | 输入负数金额 | 应拒绝 | |
| 4 | 操作人记录 | 录入后查看流水 | 流水中应记录操作管理员信息（当前已知缺陷：operatorUuid 未写入） | |
| 5 | 给未激活会员录入 | 后台对未激活会员录入积分 | 录入成功，积分可累积，用户激活后可见 | |

### 1.2 小RF积分录入

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 正常录入 | 选择会员 → 选择小朋友 → 输入积分数 → 确认 | 对应小朋友的小RF积分增加，生成 IN 流水 | |
| 2 | 会员无小朋友 | 尝试给无小朋友的会员录入小RF积分 | 提示需先绑定小朋友 | |
| 3 | 多个小朋友选对 | 会员有 A、B 两个小朋友，给 A 录入 50 分 | 仅 A 的小RF积分增加 50，B 不受影响 | |

## 二、积分余额展示（小程序端）

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 大RF积分展示 | 进入个人中心 / 积分商城 | 展示大RF积分当前可用余额 | |
| 2 | 小RF积分展示 | 已绑定小朋友 → 查看 | 按小朋友分别展示各自的小RF积分余额 | |

## 三、积分明细（小程序端）

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 查看全部流水 | 进入积分明细页 → 选"全部" | 展示收入/支出/过期全部流水，按月分组 | |
| 2 | 筛选收入 | 选"收入" | 仅展示 IN 类型流水 | |
| 3 | 筛选支出 | 选"支出" | 仅展示 OUT 类型流水 | |
| 4 | 筛选过期 | 选"过期" | 仅展示 EXPIRE 类型流水 | |
| 5 | 分页加载 | 流水较多时下拉加载 | 分页正常，无重复数据 | |
| 6 | 大RF/小RF切换 | 从商城小RF Tab 点"明细" | 应展示小RF明细（已知缺陷：当前未传 pointsType，展示大RF明细） | |
| 7 | 多小朋友小RF明细 | 多个小朋友时查看小RF明细 | 应按小朋友区分展示（已知缺陷：缺少 childUuid 传参） | |

## 四、积分更正（PC端）

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 增加更正 | 会员详情 → 积分更正 → 增加 50 分 → 填写原因 → 二次确认 | 余额增加 50，生成 IN 流水，记录更正原因 | |
| 2 | 减少更正 | 积分更正 → 减少 30 分 → 填写原因 → 二次确认 | 余额减少 30，生成 OUT 流水 | |
| 3 | 减少超过余额 | 当前余额 20，尝试减少 50 | 应拒绝，余额不允许为负 | |
| 4 | 更正后等级重算 | 大RF积分更正导致累计积分跨越等级门槛 | 触发等级重新计算 | |
| 5 | 不填原因 | 更正时不填写原因 | 应提示必填 | |

## 五、积分消费扣减

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 兑换扣减大RF | 用 100 大RF积分兑换商品 | 大RF余额减少 100，生成 OUT 流水 | |
| 2 | 兑换扣减小RF | 用 50 小RF积分兑换商品（指定小朋友 A） | 仅 A 的小RF余额减少 50，其他小朋友不受影响 | |
| 3 | 余额不足时兑换 | 大RF余额 80，尝试兑换 100 积分商品 | 提示余额不足，兑换失败 | |
| 4 | 扣减不影响等级 | 兑换扣减积分后 | 会员等级不变（等级按累计获得积分计算，非当前余额） | |
| 5 | 并发扣减 | 同一用户同时发起两笔兑换 | 仅一笔成功，另一笔余额不足失败；余额不为负 | |

## 六、积分清零

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 注册周年清零 | 会员 activatedAt 周年日到达 | 大RF + 小RF 积分全部清零，每个账户生成 EXPIRE 流水 | |
| 2 | 清零后余额 | 清零执行后查看余额 | 余额为 0 | |
| 3 | 清零后等级不降 | 清零前等级为 Lv.3 → 清零 | 等级维持 Lv.3 不降级 | |
| 4 | 清零幂等 | 同一天重跑清零任务 | 不会重复清零，检查当天已有 EXPIRE 流水则跳过 | |
| 5 | 余额已为 0 时清零 | 会员积分余额已为 0 | 跳过清零，不写多余流水 | |
| 6 | 清零提醒 | 清零前 3/2/1 个月 | 系统生成对应消息通知（当前未实现） | |

## 七、账务一致性

| # | 场景 | 操作步骤 | 预期结果 | 通过 |
|---|------|----------|----------|------|
| 1 | 余额 = 流水合计 | 执行多次录入、兑换、更正后 | 账户余额 = SUM(IN) - SUM(OUT) - SUM(EXPIRE) | |
| 2 | 余额永不为负 | 任何操作组合后 | 余额 >= 0 | |
| 3 | 流水可追溯 | 查看任意一条流水 | 能追溯到来源（录入单 / 兑换单 / 清零批次）和操作人 | |
