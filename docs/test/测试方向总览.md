# RF商城 - 测试方向总览

> 本文档按模块梳理测试重点方向、边界条件、易错点和需重点验证的场景。后续基于本文档细化为具体的测试用例。

---

## M01 用户与认证

### 核心测试方向

#### 1. 微信授权登录
- 首次微信授权能否正常获取 openid 并创建登录账号
- 已有登录账号的用户再次打开小程序，能否正常恢复登录态
- 微信授权拒绝后的降级处理（用户点击拒绝授权）
- 登录态过期后的重新鉴权流程

#### 2. 手机号绑定与匹配
- **正常匹配**：线下已建档会员（手机号已存在），首次登录绑定手机号后能否自动匹配并激活
- **自动注册**：手机号不存在时，能否自动创建会员档案 + 初始化大RF积分账户
- **冲突检测**：手机号已被其他登录账号绑定时，是否正确提示冲突并阻止绑定

#### 3. PC后台登录
- 手机号+密码登录：正确/错误密码、账号不存在、账号被停用
- 手机号+验证码登录：验证码过期、验证码已使用、验证码错误
- 普通会员尝试登录PC后台，是否提示无权限
- 管理员/超管两种角色分别登录

#### 4. 密码管理
- 管理员通过验证码修改自己的密码，修改后旧密码失效、新密码可用
- 超管重置管理员密码，重置后管理员旧密码失效
- 超管能否重置另一个超管的密码（应不允许或需特殊处理）

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 同一手机号重复绑定 | 数据冲突 | 一个手机号只能绑一个会员档案，一个登录账号只能绑一个会员档案 |
| 验证码时效 | 安全风险 | 验证码过期（expiresAt）后应不可用，used=true 后应不可重复使用 |
| 验证码类型混用 | 安全风险 | LOGIN 类型验证码不能用于 JOIN 场景，反之亦然 |
| 频繁发送验证码 | 防刷 | 是否有发送频率限制（如60秒内不可重复发送） |
| 停用账号登录 | 权限漏洞 | status=INACTIVE 的账号不应登录成功 |
| 并发绑定 | 数据一致性 | 两个登录账号同时绑定同一手机号，只能有一个成功 |
| 超管不可通过界面创建 | 越权风险 | 验证接口层面不存在创建超管的入口 |

## M02 会员信息

### 核心测试方向

#### 1. 线下建档
- 工作人员录入手机号+姓名创建会员档案，状态是否为"未激活"
- 建档时手机号重复（已存在同手机号会员）是否阻止

#### 2. 个人信息编辑
- 小程序端用户编辑姓名、生日、性别等字段，保存后数据正确
- 用户端不应有修改手机号的入口
- 后台修改手机号时，需验证新手机号是否唯一

#### 3. 小朋友管理
- 添加小朋友后，系统是否自动创建小RF积分账户
- 删除小朋友时，关联的小RF积分账户如何处理（软删？积分清零？）
- 多个小朋友各自独立的积分账户是否正确隔离
- 未绑定小朋友的用户，小程序端应隐藏小RF相关入口

#### 4. 会员列表查询
- 搜索关键词（手机号/姓名）模糊匹配正确性
- 筛选条件（会员状态、等级、建档来源）组合使用
- 导出功能的数据完整性与格式正确性

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 手机号唯一性 | 数据一致性 | 建档、修改手机号时的唯一性校验，含软删除记录的手机号污染处理 |
| 生日格式 | 数据有效性 | 非法日期（2月30日等）、未来日期是否拦截 |
| 小朋友数量上限 | 业务规则 | 是否有添加小朋友数量的上限限制 |
| 会员档案与登录账号解绑 | 数据完整性 | 是否存在解绑场景？解绑后积分、等级等数据如何处理 |
| 软删除会员再次建档 | 数据冲突 | 删除的会员手机号已做污染处理，同手机号能否重新建档 |

## M03 积分账户

### 核心测试方向

#### 1. 积分录入
- 大RF积分：根据消费金额手动录入，录入后余额增加、流水正确
- 小RF积分：选择具体小朋友录入，积分归入对应小朋友的账户
- 录入后操作日志是否完整（谁、什么时候、给谁、多少积分）
- 录入后是否触发等级重算（仅大RF积分）

#### 2. 积分消费（兑换扣减）
- 余额充足时扣减正确，流水记录正确
- 余额不足时拒绝扣减并提示
- 小RF积分扣减时必须指定小朋友，不可跨小朋友合并消费
- 积分扣减不影响会员等级（等级看累计获得，不看余额）

#### 3. 积分更正
- 更正通过反向流水实现，原流水不被修改
- 正向更正（补发）后余额增加，负向更正（扣回）后余额减少
- 负向更正后余额是否可能为负数（应阻止）
- 更正大RF积分后是否触发等级重算（正向可能升级，负向可能降级）
- 更正时必须填写原因

#### 4. 积分清零
- 清零时间计算：注册周年日是否准确（考虑闰年2月29日的情况）
- 大RF和小RF同时清零，各自独立写流水
- 清零后等级不降级
- 清零前3/2/1个月的提醒是否准确发出
- 已清零的不重复清零（幂等）

#### 5. 积分流水
- 流水记录完整性：类型、方向、金额、原因、时间、操作人
- 余额由流水累计得出，不能手动修改余额字段

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 并发扣减 | 超扣 | 两个兑换请求同时发起，积分只有一份够用，应只有一个成功 |
| 积分余额为负 | 数据异常 | 任何场景余额都不能为负数 |
| 幂等控制 | 重复入账 | 录入、兑换、清零都要有业务唯一号，重复请求不重复入账 |
| 录入0积分或负数积分 | 数据有效性 | 录入积分数应为正整数，边界校验 |
| 大量积分录入 | 数值溢出 | Int 类型上限，极端情况下累计积分是否溢出 |
| 清零时间——2月29日注册 | 日期边界 | 闰年2月29日注册的会员，非闰年周年日如何处理 |
| 清零日恰好有退还积分 | 执行顺序 | 超时订单退积分和清零在同一天执行，退回的积分是否也被清零 |
| operatorUuid 未写入流水 | 审计缺失 | 文档已标注操作1/2/3的 operatorUuid 未写入 PointsLedger，需修复验证 |
| 解绑小朋友积分处理 | 数据丢失 | 解绑时小RF积分账户软删，但没写清零流水，需评估 |

## M04 会员等级

### 核心测试方向

#### 1. 等级自动计算
- 累计大RF积分从 0 → 6000，等级从 Lv.1(考拉) 升到 Lv.2(小熊)
- 逐级跨越每个门槛：0/6000/12000/30000/50000
- 恰好到达门槛值时是否正确升级（如恰好 6000 分）
- 门槛值 -1（如 5999 分）时不应升级

#### 2. 等级升降规则
- 积分清零后等级不降级
- 积分更正（负向）导致累计积分低于当前等级门槛时，等级是否降级（文档说可能降级）
- 兑换扣减积分不影响等级（扣的是可用余额，不是 totalEarned）

#### 3. 等级持久化
- `MemberProfile.levelUuid` 在录入大RF积分后是否正确更新
- 后台建档时手动指定等级，是否正确保存
- 后台手动修改等级，是否可覆盖系统计算值

#### 4. 等级权益展示
- 各等级权益列表配置后，小程序端正确展示
- 等级图标、名称显示正确

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 连续多次录入触发连跳 | 跳级 | 一次录入大量积分直接从 Lv.1 跳到 Lv.4，是否正确处理 |
| 等级门槛边界 | 计算错误 | 恰好 6000、5999、6001 分别对应什么等级 |
| 小RF积分录入不触发升级 | 误触发 | 小RF积分不参与等级计算，录入小RF后等级不应变化 |
| 后台手动设等级与系统计算冲突 | 数据不一致 | 手动设为Lv.5但累计积分只有100，后续录入积分后等级会变回Lv.1吗 |
| 清零后再录入积分 | 等级异常 | 清零后累计积分不变（清零的是可用余额），录入新积分 totalEarned 继续累加 |

## M05 商品管理

### 核心测试方向

#### 1. 商品状态流转
- 草稿 → 仓库 → 已上架 → 已下架 → 已上架（正常流转）
- 草稿不能直接上架（需先进入仓库）
- 已上架状态可编辑信息但不改变状态
- 只有"已上架"商品在小程序可见

#### 2. 商品池隔离
- 大RF商品和小RF商品互不混用
- 创建后积分类型不可更改
- 未绑定小朋友的用户隐藏小RF Tab

#### 3. 规格与SKU
- 无规格商品自动生成默认SKU
- 多规格组笛卡尔积正确生成SKU
- 主规格组规格图片切换
- 新增/删除规格值后SKU同步更新
- SKU库存独立管理

#### 4. 库存与限购
- 单个SKU库存为0显示"已兑完"
- 所有SKU库存为0，商品整体显示"已兑完"
- 限购规则：每月限兑N件，所有SKU合计计算
- 限购跨月重置

#### 5. 券类商品
- `isCoupon=true` 的商品兑换后额外生成 UserCoupon
- 券模板关联正确（面额/折扣、有效期、使用规则）
- 券状态流转：UNUSED → USED（手动核销）/ EXPIRED（定时任务过期）
- 过期券不退积分

#### 6. 商品/分类删除
- 商品删除后同步清理SKU、购物车关联、规格关联
- 删除商品后历史订单不受影响
- 有关联商品的分类不可删除

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 商品下架后购物车中的处理 | 用户体验 | 已加入购物车的商品被下架，结算时如何处理 |
| 库存并发扣减 | 超卖 | 多人同时兑换最后1件商品，只能有一个成功 |
| 限购计算跨月 | 规则错误 | 月底和月初的限购计数是否正确重置 |
| 规格值为空 | 数据异常 | 创建规格组但不添加规格值时的处理 |
| 主规格组删除 | 图片丢失 | 删除已设为主规格组的规格组后，图片切换逻辑如何降级 |
| 券有效期——expiresAt 和模板 validTo | 优先级混乱 | UserCoupon.expiresAt 优先，为空时回退到 CouponTemplate.validTo |
| 券类商品下架后已持有的券 | 状态不一致 | 商品下架不影响已发放的券，用户仍可查看和使用 |
| 商品积分价格为0 | 白嫖 | 是否允许0积分商品，若允许需要注意限购 |
| 商品图片为空 | 展示异常 | 商品必填图片，校验是否严格 |

## M06 兑换交易（含核销）

### 核心测试方向

#### 1. 购物车
- 大RF和小RF商品分组展示
- 小RF商品必须关联幼崽，可切换不同幼崽
- 积分不足不可加入购物车
- 商品下架/删除后购物车中对应项的处理

#### 2. 兑换下单
- 事务完整性：创建订单 → 扣积分 → 扣库存 → 写流水 → 清购物车 → 生成核销码，任一环节失败全部回滚
- 混合下单：大RF+小RF商品同时结算，分别扣减对应积分
- 同一订单内小RF商品关联不同幼崽，各自扣减对应幼崽积分
- 核销码生成：8位数字码，订单级+明细级各一个，互不重复

#### 3. 核销流程
- 扫码核销：扫描明细级核销码，正确识别对应订单和明细行
- 输码核销：手动输入8位数字码核销
- 分次核销：同一明细行支持多次核销直到全部核销
- 全部核销后订单状态变为 VERIFIED
- 核销不可撤销

#### 4. 超时处理
- PENDING_VERIFY 超10天：自动取消，退还全部积分和库存
- PARTIAL_VERIFIED 超10天：退还未核销部分的积分和库存，已核销部分保留，订单变 VERIFIED
- 不支持用户主动取消

#### 5. 立即兑换（跳过购物车）
- 大RF商品：选规格→结算→确认
- 小RF商品：选幼崽→结算→确认

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 并发下单扣积分 | 超扣 | 积分只够一单但同时提交两单，只有一单成功 |
| 并发下单扣库存 | 超卖 | 库存只剩1件但同时两人下单，只有一单成功 |
| 核销码重复 | 核销错乱 | 8位数字码在全系统范围内是否保证唯一 |
| 已过期核销码尝试核销 | 安全风险 | 超过有效期的核销码应拒绝核销 |
| 部分核销后剩余项过期 | 状态混乱 | 明细行部分核销 + 超时退还的积分计算和库存恢复是否正确 |
| 购物车中商品积分变动 | 价格不一致 | 加入购物车后商品积分价格被修改，结算时用哪个价格 |
| 下单时积分刚好够/差1分 | 边界值 | 积分余额恰好等于所需积分时能否下单成功 |
| 大量明细行的订单 | 性能 | 一次下单包含很多件商品时事务性能 |
| 同一SKU不同幼崽加购 | 购物车去重 | userUuid+skuUuid+childUuid 唯一约束，同SKU不同幼崽应为不同行 |
| 核销码展示安全性 | 信息泄露 | 核销码二维码+数字码只在相关用户页面可见 |

## M09 消息通知

### 核心测试方向

#### 1. 消息触发完整性
- 每种消息类型（共14种）是否在对应场景正确生成
- 实时触发类（积分录入、兑换、核销、升级、激活）：操作后立即出现消息
- 定时触发类（清零提醒、超时取消、券过期提醒）：定时任务执行后正确生成

#### 2. 消息内容正确性
- 模板变量替换正确：金额、余额、商品名、日期等
- 消息优先级标记正确（重要/普通）
- linkUrl 跳转路径正确且参数完整

#### 3. 消息状态
- 未读消息显示红点
- 点击消息后标记已读
- 所有消息已读后红点消失
- 消息列表按时间倒序排列

#### 4. 防重复
- POINTS_EXPIRY_WARN：同一会员+同一清零周期+同一档位不重复
- VERIFY_CODE_EXPIRING：同一订单只发一次
- COUPON_EXPIRY_WARN：同一张券只发一次

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 消息关联用户的准确性 | 错发 | 消息的 userUuid 是否正确（特别是小RF积分录入，消息应发给父母而非小朋友） |
| 积分更正消息的 delta 显示 | 用户困惑 | 正向更正和负向更正的文案是否区分清楚 |
| 定时任务失败后重跑 | 重复消息 | 幂等检查是否有效，重跑不产生重复消息 |
| 大量未读消息 | 性能 | 用户长期不看消息，列表加载和红点计算性能 |
| linkUrl 跳转到不存在的页面 | 体验 | 关联的订单/券被删除后点击消息的处理 |
| bizUuid 拼接格式 | 防重失效 | bizUuid 的拼接规则是否一致，格式不一致会导致防重失败 |

## M10 数据统计

### 核心测试方向

#### 1. 工作台概览（Dashboard）
- totalMembers 统计准确（排除软删除）
- todayNew 是否严格按"今日0点"计算（注意时区问题）
- pendingActivationCount 是否只统计未激活状态
- pendingVerifyOrders 是否只统计待核销订单
- stockAlertCount 阈值为10，验证边界（10/11/9/0）

#### 2. 兑换统计
- totalOrders 排除软删除
- totalPointsConsumed 计算大RF+小RF之和
- topProducts TOP10 排序正确性
- 核销率前端计算 verifiedOrders/totalOrders，分母为0的处理

#### 3. 会员统计
- 等级分布按 levelUuid 分组，等级为空的会员如何处理
- withChildren 统计是否排除已解绑（软删除）的小朋友

#### 4. 积分统计
- totalIssued 仅统计 IN 流水
- totalConsumed 仅统计 OUT 流水
- currentBalance 统计所有积分账户余额之和
- 大RF/小RF 分别统计，不混淆
- 使用率前端计算 totalConsumed/totalIssued，分母为0的处理

#### 5. 库存预警
- stockAlerts 条件：在售SKU且 0 < stock <= 10
- soldOutCount：在售SKU且 stock <= 0
- 预警列表排序：按库存升序

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 时区问题 | 统计偏差 | "今日新增"是否按服务器时区/东八区正确计算 |
| 软删除数据参与统计 | 数据失真 | 所有统计是否正确排除了 deletedAt 不为空的记录 |
| 大数据量下的统计性能 | 接口超时 | 数据量大时统计查询是否有索引支撑、是否需要缓存 |
| 分母为零 | 前端崩溃 | 核销率、使用率等衍生指标分母为0时前端是否有防护 |
| 库存预警阈值硬编码 | 灵活性 | STOCK_ALERT_THRESHOLD=10 硬编码，后续是否需要可配置 |
| 统计口径一致性 | 数据矛盾 | Dashboard 和 Overview 中重复字段（如 totalMembers）数值是否一致 |

## M13 定时任务

### 核心测试方向

#### 1. 整体调度
- `POST /cron/daily` 接口可正常调用并返回执行结果
- 子任务按顺序依次执行
- 单条失败不影响其他记录继续执行
- 返回格式正确：每个子任务的 processed/skipped/failed 计数

#### 2. 子任务1：超时订单处理
- 1a 完全未核销：状态改CANCELLED、积分全额退还、库存全部恢复、写消息通知
- 1b 部分核销：仅退未核销部分的积分和库存、已核销部分保留、订单变VERIFIED
- 幂等：重复执行不产生多余退还

#### 3. 子任务2：注册周年积分清零
- 周年日计算准确（activatedAt 的月日匹配）
- 大RF和小RF同时清零
- 余额为0的账户跳过（不写无意义的清零流水）
- 清零后等级不降级

#### 4. 子任务3：积分清零提醒
- 90/60/30天三个档位分别提醒
- 余额为0的会员不提醒
- 同一档位不重复提醒（幂等）

#### 5. 子任务4：核销码即将过期提醒
- 到期前2天发送提醒
- 覆盖 PENDING_VERIFY 和 PARTIAL_VERIFIED
- 同一订单只发一次

#### 6. 子任务5：优惠券即将过期提醒
- 到期前3天发送提醒
- expiresAt 优先，回退到 CouponTemplate.validTo
- 同一张券只发一次

#### 7. 子任务6：过期优惠券状态更新
- UNUSED 且过期 → 自动改 EXPIRED
- 已为 USED/EXPIRED 的不处理

### 边界与易错点

| 场景 | 风险 | 验证要点 |
|------|------|----------|
| 执行顺序依赖 | 逻辑错误 | 子任务1退积分 → 子任务2清零：如果同一天退还+清零，退回的积分是否也被清零 |
| 跨时区 / 跨天边界 | 时间判断错误 | 凌晨执行时"今天"的判断是否正确（UTC vs 东八区） |
| 大量待处理数据 | 超时 | FC 函数超时 300秒，数据量大时是否能在限时内完成 |
| 手动补跑幂等性 | 重复处理 | 一天内多次调用 /cron/daily，结果应与一次调用一致 |
| 事务内部分失败 | 数据不一致 | 单个订单处理中途失败，该订单的积分退还和库存恢复是否一致回滚 |
| 清零日期计算——闰年 | 日期bug | 2月29日注册的会员，非闰年的周年日是2月28日还是3月1日 |
| 清零提醒与实际清零不一致 | 用户投诉 | 提醒和清零使用同一套日期计算逻辑，避免"说好了3月1日清零结果2月28日就清了" |
| 过期券提醒发送后券未过期 | 时间窗口 | 提醒在子任务5，状态更新在子任务6，确保先提醒再改状态 |
| 端点无鉴权 | 安全风险 | 当前 /cron/daily 无鉴权，任何人可触发，需尽快加密钥或IP白名单 |

## 跨模块集成测试重点

以下场景涉及多个模块协同，容易在模块交界处出问题：

### 1. 完整用户生命周期
```
线下建档(M02) → 小程序授权登录(M01) → 绑定手机号匹配激活(M01+M02)
→ 录入积分(M03) → 等级升级(M04) → 浏览商品(M05) → 兑换下单(M06)
```

```
→ 浏览小RF商品(M05) → 指定小朋友兑换(M06) → 核销(M06)
→ 小RF积分清零(M13)
```

### 3. 券类商品完整链路
```
创建券类商品+券模板(M05) → 上架(M05) → 用户兑换(M06)
```

- 统计接口返回的数据（M10）与实际数据库数据交叉验证
- 兑换统计的积分消耗总量 = 积分流水中兑换支出的总和
- 会员总数与会员列表记录数一致

## 通用测试关注点

### 数据有效性
- 所有必填字段的空值检查
- 字符串字段的长度上限（手机号11位、验证码4-6位、核销码8位等）
- 数值字段的范围检查（积分>0、库存>=0、价格>0）
- 枚举字段的非法值检查

### 软删除
- 软删除后列表查询是否正确过滤
- 软删除记录的 unique 字段是否做了污染处理（如手机号加后缀）
- 软删除后关联数据的处理（如删除商品后订单中的商品信息保留）

### 分页与排序
- 每个列表接口的分页参数校验（page<1、limit<0、limit极大值）
- 排序字段注入风险
- 空列表返回 `{ list: [], pagination: { page, limit, total: 0 } }`

### 接口安全
- 所有接口是否有 JWT/登录态校验
- PC后台接口是否有角色权限校验
- 用户A不能操作用户B的数据（水平越权）
- 普通用户不能调用管理员接口（垂直越权）
- 定时任务接口的鉴权（目前无，需尽快补充）
